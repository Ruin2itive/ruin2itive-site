üß™ Manual Testing Instructions

Prerequisites:

  1. Start a local web server: python3 -m http.server 8000
  2. Open 3 different browser windows/tabs:
    • Window 1: Regular browser
    • Window 2: Incognito/Private mode
    • Window 3: Different browser or another incognito window
  3. Navigate to http://localhost:8000/chat.html in each window

Test Scenario 1: Basic Message Broadcasting

Steps:

  1. Join chat as "Alice" in Window 1
  2. Join chat as "Bob" in Window 2
  3. Join chat as "Charlie" in Window 3
  4. Wait for all users to connect (status indicator should be green)
  5. Send "Hello from Alice" from Window 1
  6. Send "Hello from Bob" from Window 2
  7. Send "Hello from Charlie" from Window 3
‚úÖ Expected Result:
  • All three messages should appear in all three windows
  • No duplicate messages should appear
  • Messages should appear in the same order across all windows

Test Scenario 2: Late Joiner

Steps:

  1. With Alice, Bob, and Charlie already connected (from Scenario 1)
  2. Alice sends "Message 1"
  3. Bob sends "Message 2"
  4. Open a 4th window and join as "Dave"
  5. Dave should see recent message history
  6. Dave sends "Hello everyone!"
  7. Charlie sends "Welcome Dave!"
‚úÖ Expected Result:
  • Dave receives recent message history upon joining
  • Dave's message appears in all windows
  • Charlie's message appears in all windows including Dave's

Test Scenario 3: Rapid Messaging

Steps:

  1. Using the connected users from previous scenarios
  2. Quickly send 3-4 messages from different users in rapid succession
  3. Example:
    • Alice: "Message A1"
    • Bob: "Message B1"
    • Charlie: "Message C1"
    • Alice: "Message A2"
‚úÖ Expected Result:
  • All messages appear in all windows
  • No messages are lost
  • No duplicate messages appear
  • Message order is consistent across all windows

Test Scenario 4: Console Verification

Steps:

  1. Open browser Developer Tools (F12) in one of the windows
  2. Go to the Console tab
  3. Send a message from a different window
  4. Observe the console logs
‚úÖ Expected Result:

You should see logs like:

  • [DATA] Received data from peer - Message received
  • [DATA] Duplicate message detected, skipping - Deduplication working
  • No error messages in red

üîç What Was Fixed

Problem Before Fix:

In a mesh network with Users A, B, and C:

  • User B sends a message
  • User A receives it (via incoming connection) ‚úì
  • User C receives it (via outgoing connection) ‚úì
  • User A relays to User C ‚úì
  • User C does NOT relay ‚ùå (outgoing connections didn't relay)
  • Result: Messages from some users don't reach everyone

Solution After Fix:

Now with the fix:

  • User B sends a message with unique ID
  • User A receives it and relays to others ‚úì
  • User C receives it and relays to others ‚úì
  • Duplicate messages are detected and filtered ‚úì
  • Result: All messages reach all users exactly once! üéâ

üìä Technical Details

  • Message ID: Each message gets a unique ID: randomString-timestamp
  • Deduplication: Each peer tracks up to 200 seen message IDs
  • Relay Logic: Both incoming and outgoing connections relay messages to all other peers
  • Memory Management: Automatic cleanup keeps only recent message IDs
  • Security: All changes verified with CodeQL scanner (0 vulnerabilities)
